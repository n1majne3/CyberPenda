# Use flat semantic Blackboard change batches

Runtime and operator Blackboard writes use one atomic `Semantic Change Batch` with an idempotency key and a flat list of `create`, `update`, `transition`, `relate`, `unrelate`, `merge`, or `supersede` changes. Records and relationship endpoints are addressed only by project-wide Blackboard Keys; a create names its record type and closed type-specific record, and later changes in the same batch may immediately reference that chosen key. Existing state changes carry current versions. The protocol removes operation IDs, immutable node/edge IDs, nested node references, generic property maps, caller-supplied Project/context, and storage-oriented mutation kinds while preserving atomic validation, optimistic concurrency, and replay-safe retries.

Successful responses return the resulting graph revision and only changed Blackboard Keys or relationship triples with their versions; mutation IDs/sequences, timestamps, hashes, internal IDs, and actor/context data are omitted. Version conflicts name the key, expected/current versions, and the next semantic action. When no unseen external change exists, the response reports the atomically updated Working Snapshot path/revision without embedding the full graph. When another Task changed shared knowledge since the caller's last acknowledged revision, the response instead piggybacks the complete current Runtime Blackboard Snapshot and a concise explanation, then acknowledges that external change.
