CyberPenda 仓库未引入 Viper、pflag/cobra 等通用配置框架，而是采用 Go 标准库 `flag` + 环境变量直读的最简模式，配合运行时目录结构完成“进程级配置”与“任务/Provider 级配置”的分层。

## 1. 系统概览：两层配置模型

- **进程级配置（pentestd / pentestctl）**：通过命令行参数解析，每个参数都提供对应的 `PENTEST_*` 环境变量作为覆盖源；无 YAML/TOML/JSON 配置文件。
- **任务/Provider 级配置**：在沙箱隔离的 `runs/<task-id>/runtime-home/<provider>/config.toml` 下由 Runner 投影生成，供被调用的 Provider 进程读取。

## 2. 进程级配置入口与约定

### pentestd（守护进程）

`cmd/pentestd/main.go` 使用 `flag.StringVar` 声明所有可配置项，并通过自定义 `envOr(key, fallback)` 实现“环境变量优先于默认值”的覆盖策略：

| 标志 | 环境变量 | 默认值 | 说明 |
|------|----------|--------|------|
| `-addr` | `PENTEST_LISTEN_ADDR` | `127.0.0.1:8787` | HTTP/MCP 监听地址 |
| `-db` | `PENTEST_DB` | `pentest.db` | SQLite 数据库路径 |
| `-runtime-root` | `PENTEST_RUNTIME_ROOT` | 空（自动推导） | 任务运行根目录 |
| `-artifact-root` | `PENTEST_ARTIFACT_ROOT` | 空（自动推导） | Evidence 制品根目录 |
| `-evidence-source-roots` | `PENTEST_EVIDENCE_SOURCE_ROOTS` | 空 | 允许保留 Evidence 的本地根路径列表（逗号分隔） |
| `-sandbox-image` | `PENTEST_SANDBOX_IMAGE` | `ghcr.io/n1majne3/cyberpenda-sandbox:latest` | 沙箱容器镜像 |
| `-container-cli` | `PENTEST_CONTAINER_CLI` | `docker` | 容器 CLI（docker 或 podman） |
| `-runtime-plugin-dirs` | `PENTEST_RUNTIME_PLUGIN_DIRS` | 空 | 可信 runtime plugin manifest 目录（逗号分隔） |
| `-runtime-extension-dirs` | `PENTEST_RUNTIME_EXTENSION_DIRS` | 空 | 可信 runtime extension manifest 目录（逗号分隔） |
| `-auth-token` | `PENTEST_AUTH_TOKEN` | 空 | 非 loopback 绑定时的 Bearer token |

这些参数随后组装为 `daemon.Config` 结构体传入 `daemon.NewServer`，并在启动时执行安全校验：若监听地址非 loopback 且未设置 `AuthToken`，直接拒绝启动。

### pentestctl（离线 CLI）

`internal/pentestctl/pentestctl.go` 同样基于 `flag.NewFlagSet`，支持 `--db`、`--api`、`--token` 三个全局标志，以及子命令 `blackboard v2 inspect/backup/migrate/verify` 各自的标志。此外还从以下环境变量加载上下文：

- `PENTEST_PROJECT_ID`
- `PENTEST_TASK_ID`
- `PENTEST_CONTINUATION_ID`
- `PENTEST_API_URL`

## 3. 任务/Provider 级配置：Runner 投影生成的 config.toml

Runner 在每次 Launch 时为每个 Provider 在 `runs/<task-id>/runtime-home/<provider>/` 下投影一份只读的 `config.toml`，内容包含：

- 模型端点、API Key（来自 Runtime Profile + Credential Binding）
- MCP 服务器清单与注入的环境变量（如 `PENTEST_MCP_URL`）
- 其他 Provider 特定的字段（例如 Codex 的 `base_url`、`model`）

该文件路径由 `internal/runner/blackboard_v2_layout.go` 与 `internal/runtimeplugin/builtin.go` 中的 `ConfigPath: "runtime-home/codex/config.toml"` 约定确定，并由 `internal/runner/projection.go` 负责写入。

## 4. 架构与设计决策

- **零外部依赖**：不引入 Viper/HCL/YAML 解析器，降低二进制体积与攻击面。
- **显式优于隐式**：所有可配置项均通过 flag 暴露，不存在“魔法键名”；环境变量命名统一以 `PENTEST_` 前缀。
- **分层职责清晰**：
  - 进程级配置 → `flag` + `os.Getenv` → `daemon.Config`
  - 任务级配置 → Runner 投影 → 沙箱内 Provider 进程自读 `config.toml`
- **安全默认**：非 loopback 绑定强制要求 `PENTEST_AUTH_TOKEN`；Credential 注入需显式开启 `PENTEST_ALLOW_COMMAND_CREDENTIALS`。

## 5. 开发者应遵循的规则

1. **新增进程级配置项**：在 `cmd/pentestd/main.go` 中用 `flag.StringVar` 声明，并配套添加 `PENTEST_<NAME>` 环境变量覆盖；同时在 `daemon.Config` 中添加字段并在 `NewServer` 中做必要校验。
2. **新增任务/Provider 配置字段**：在 `internal/runner/projection.go` 的投影逻辑中追加写入到 `config.toml`，并确保对应 Provider 能读取该字段。
3. **避免引入新的配置文件格式**：进程级配置坚持 flag/env 模式，不要新增 YAML/TOML 形式的进程级配置文件。
4. **敏感信息走 Credential 系统**：API Key 等敏感值不应硬编码进进程级配置，而应通过 Runtime Profile + Credential Binding 注入到任务级 `config.toml`。
