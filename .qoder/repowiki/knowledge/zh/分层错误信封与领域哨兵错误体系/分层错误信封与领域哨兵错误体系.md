本仓库采用「语义层错误信封 + 领域哨兵错误 + HTTP 状态映射」的分层错误处理策略，贯穿 Blackboard v2 语义服务、Project Interface 协议边界、运行时 Provider Session 以及守护进程 HTTP 适配层。

## 1. 系统/方法概述
- **语义层错误信封**：internal/blackboardv2/service.go 定义 Error 结构体（code/message/path/retryable/details），并通过 semanticError(code, message, path, details) 构造，作为所有领域操作返回的稳定错误载体。
- **Project Interface 协议错误**：internal/projectinterface/errors.go 定义更丰富的 Error 信封（含 protocol_version、op_id、request_id、retryable、details），并集中声明所有协议级错误码常量（如 ErrCodeInvalidRequest、ErrCodeStorageBusy、ErrCodeSnapshotUnavailable 等）。提供 ValidationError、InternalError、persistenceError 工厂以及 AsError 提取器。
- **HTTP 状态映射**：internal/daemon/blackboard_v2_http.go 中的 blackboardV2HTTPStatus 将语义错误 code 精确映射到 HTTP 4xx/5xx（如 storage_busy → 503、version_conflict → 409、authority_denied → 401/403、invalid_schema → 400、closed_continuation → 410）。
- **领域哨兵错误**：各模块以包级 var ErrXxx = errors.New(...) 暴露可被 errors.Is 匹配的哨兵错误，例如：
  - internal/blackboardmigration/service.go: ErrMigrationBlocked、ErrCutoverVerificationFailed、ErrUnsupportedMigrationKind
  - internal/blackboardv2/continuity.go: ErrLaunchPinIntegrity
  - internal/project/project.go: ErrNotFound、ErrInvalidKind、ErrMissingName
  - internal/adapters/adapters.go: ErrSecretRedacted
  - internal/runtime/provider_session.go: ErrProviderSessionRequestConflict、自定义类型 ProviderSessionRequestConflictError、UnsupportedProviderSessionCapabilityError、ProviderSessionOperationError（实现 Unwrap）
  - internal/runtime/session_bridge.go: SandboxBridgeRPCError
  - internal/runtimeprofile/custom_args_conflict.go: ErrCustomArgConflict
- **无 panic/recover**：全仓未发现任何 panic 或 defer recover 使用，错误一律通过返回值传播。

## 2. 关键文件与包
- internal/projectinterface/errors.go — 协议级错误信封、错误码常量、AsError 提取器
- internal/blackboardv2/service.go — 语义层 Error 结构与 semanticError 构造器
- internal/daemon/blackboard_v2_http.go — 语义错误 → HTTP 状态码映射
- internal/blackboardmigration/service.go、rebuild_v2.go、migrate_v2.go — 迁移域哨兵错误
- internal/runtime/provider_session.go — Provider Session 专用错误类型与哨兵
- internal/pentestctl/blackboard_v2.go — CLI 侧内部错误包装器 blackboardV2InternalError

## 3. 架构与约定
- **错误向上冒泡不吞掉**：领域函数返回 *blackboardv2.Error 或哨兵错误，上层适配器/HTTP 处理器负责分类与响应编码，不在业务层直接写日志或打印。
- **可重试性显式标记**：Retryable: true 的错误（如 storage_busy）统一走 503，客户端据此实施指数退避；非 retryable 的语义校验错误走 4xx。
- **细节透传**：Details map[string]any 携带结构化诊断信息（如 reason、next_action、attempts 列表），测试中通过 assertV2ErrorEnvelope 断言其存在。
- **错误解包**：自定义错误类型实现 Is / Unwrap（如 ProviderSessionRequestConflictError.Is、ProviderSessionOperationError.Unwrap），配合 errors.Is 做跨层匹配。
- **持久化异常归一化**：projectinterface.persistenceError 检测 SQLite busy/locked 消息并改写为 ErrCodeStorageBusy，屏蔽底层驱动差异。

## 4. 开发者应遵循的规则
1. 领域函数只返回错误：不要 log.Fatal、不要 panic，用 fmt.Errorf("%w", ...) 包裹底层错误，或用 semanticError 构造语义错误。
2. 对外暴露的协议错误必须走 projectinterface 信封：在 HTTP/MCP 边界使用 ValidationError / InternalError / persistenceError 构造，确保包含 protocol_version、op_id、request_id。
3. 需要被调用方条件分支的错误定义为包级哨兵 var ErrXxx，并在文档注释说明触发场景，供 errors.Is 判断。
4. HTTP 层只做映射不做分类：依赖 blackboardV2HTTPStatus 把 code 转为状态码，新增错误码时同步更新该 switch。
5. 避免泄露敏感信息：对可能包含凭据的错误使用 ErrSecretRedacted 或类似模式，确保 Error() 输出不包含明文 secret。
6. 测试覆盖错误路径：使用 assertV2ErrorEnvelope 断言 code/status/details，并用 errors.Is 验证哨兵错误可被正确识别。
