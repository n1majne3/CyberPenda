- Semantic errors are returned through a typed `*blackboardv2.Error` with a machine-readable `Code` (e.g. `semantic_validation`, `project_kind_mismatch`, `version_conflict`, `authority_denied`) rather than Go errors, so tests assert via `errors.As` and `isSemanticCode`.
- Validation helpers return `semanticError(code, message, path, details)` carrying a dot-notation JSONPath to the offending field inside the ChangeBatch, which tests check both the code and the `Path`/`Details` fields.
- Read-only queries open a `ReadOnly: true` transaction via `s.db.BeginTx(ctx, &sql.TxOptions{ReadOnly: true})` and defer `tx.Rollback()` to guarantee no accidental writes on snapshot/read paths like `CTFSolvedState`.
- Change handlers follow a uniform signature `(ctx, tx, projectID, revision, index, change, now) (int, string, int, bool, error)` returning new revision, affected key, version, and a boolean identity flag, letting the caller short-circuit when the patch is a no-op.
- Tests construct scenarios through `service.Apply`/`ApplyForContinuation` with `Schema: "semantic-change-batch/v2"` and `IdempotencyKey`, then verify side effects via `ReadCurrent`, `ReadHistory`, and `RuntimeSnapshot`, using `assertContractJSON` to pin DTO shapes.
