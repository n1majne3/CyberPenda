Three sibling methods on `Service` in package `blackboardv2`, each owning one lifecycle or diagnostic concern:
- `finish.go`: `FinishContinuation` is the write path. It serializes access via `snapshotMu` + `writeMu`, runs inside a single `sql.Tx`, validates ownership through a join of `task_continuations → tasks → blackboard_v2_continuation_pins`, enforces idempotency by storing a SHA-256 request hash alongside the result in `blackboard_v2_continuation_finishes`, rejects non-terminal Attempts and pending Evidence writes, updates `last_acknowledged_revision`/`working_snapshot_bytes` in `blackboard_v2_continuation_state`, materializes the closed Working Snapshot via `materializeWorkingSnapshot`, then marks the Continuation `completed`. A `FinishFailureInjector` hook lets tests inject failures at `before_commit`.
- `resume.go`: `InterruptedAttemptCheckpoints` is a read-only allowlist query against `blackboard_v2_record_history` filtered to the latest version per key with status `interrupted`, bounded by `MaxInterruptedAttemptCheckpoints = 20`, and gated behind `isTerminalContinuationStatus` + `reconciliation == completed`.
- `health.go`: `ProjectSemanticHealth` opens a read-only transaction, loads all records (entity/objective/attempt/fact/finding/solution/evidence) plus every relationship row, attempts to project the canonical Runtime Snapshot first and falls back to a health-safe diagnostic encoding when projection fails, then emits anomalies for attention budget thresholds, dangling/invalid/cyclic relationships, stranded open work, missing evidence, unresolved contradictions, and redirect integrity — finally collapsing them into a deterministic `SemanticHealth` DTO with an overall `HealthStatus` derived from the worst anomaly severity.

Dependency direction: these files depend only on shared helpers (`semanticError`, `decodeStoredRecord`, `projectRuntimeSnapshot`, `currentRevisionOrZero`, `ensureProjectExists`) and the `blackboardv2grammar` package; they do not import each other.
