Two cooperating concerns live in package `blackboardv2`:
- `projection.go` defines the read-only `RuntimeSnapshotProjection` / `RuntimeSnapshotMeasurement` types and the `ProjectRuntimeSnapshot` / `ProjectRuntimeSnapshotTx` methods that marshal a `RuntimeSnapshot` to JSON, then classify its size against fixed thresholds (`attentionTargetTokens=16K`, `attentionWarningTokens=32K`, `attentionRequiredTokens=64K`) into an `AttentionBudgetState`. This is purely a view layer — it never mutates state.
- `merge.go` implements the write path for `Op: "merge"` (and `unrelate`) inside change-batch transactions. `applyMerge` validates type/project/version constraints, optionally re-applies a `canonical_record` patch plus `Clear` fields via `applyUpdateRecord`, archives the source record + all its relationships into `_history`, deletes the source row from `blackboard_v2_records`, inserts a `blackboard_v2_key_redirects` row, and rewrites every relationship whose `from`/`to` pointed at the source so it now points at the canonical key (skipping self-links and deduplicating collisions). `duplicateFingerprint` uses SHA-256 over NFKC-normalized, punctuation-stripped text per type to enforce the pre-approved similarity requirement; `normalizeMergeLocator` handles host/endpoint normalization.
- `merge_service_test.go` is an end-to-end test suite exercising rejection paths (self-merge, cross-type, Current Work, stale versions, unapproved similarity, cross-project), idempotent replay, redirect persistence across process restarts, concurrent merge safety, Evidence/Solution/Fact/Finding merge variants, and redirected `unrelate`/`RetainEvidenceForContinuation`.

Dependency direction is one-way: tests depend on both files; projection has no merge dependencies; merge depends on shared helpers (`loadCurrentRecordDirect`, `incrementRevision`, `validateKey`, `semanticError`, etc.) defined elsewhere in the package.
