Three cooperating Go files implement the Blackboard v2 service seam:
- `service.go` defines the core `Service` struct (backed by `*store.DB`) plus all DTOs (`ChangeBatch`, `Change`, `Record`, `RuntimeSnapshot`, `CurrentDetail`, etc.) and their closed-schema `UnmarshalJSON`/`MarshalJSON` validators. The single public entry point `Apply` (and `ApplyForContinuation`) serializes writes via `writeMu`, enforces idempotency through a request-hash lookup before starting a transaction, dispatches each `Change.Op` (create/update/relate/unrelate/transition/supersede/merge) into typed handlers, and returns a `ChangeResult` with changed record tuples and a `WorkingSnapshot` pointer.
- `continuity.go` adds `ContinuityService` (depends on `*Service`, `*task.Service`, `projectinterface.GrantStore`) for launching offline Runtime sessions: `CreateContinuation` projects an immutable Launch Pin, issues a grant token, materializes the initial Working Snapshot under `<runtimeRoot>/<taskID>/workdir/.pentest/blackboard.json`, and exposes `SynchronizeContinuation` / `CaptureTrustedSynchronization` / `ClaimTrustedSynchronization` that atomically advance `blackboard_v2_continuation_state.last_acknowledged_revision` while publishing exact snapshot bytes to disk, with pre-commit publish + rollback restore for crash safety.
- `checkpoint.go` provides the lightweight `CheckpointAttemptRequest` adapter used by Runners to version an owned Attempt's compact summary; it reuses the same `apply` pipeline so checkpoints share history/idempotency/Working-Snapshot semantics.

Dependency direction is one-way: `continuity.go` → `service.go`; `checkpoint.go` → `service.go`. Both layers reach `internal/store`, `internal/task`, and `internal/projectinterface` but never vice versa. All DB access goes through raw SQL in these files; schema constants (`changeBatchSchema`, `snapshotSchema`, etc.) are the only contract surface exposed to callers.
