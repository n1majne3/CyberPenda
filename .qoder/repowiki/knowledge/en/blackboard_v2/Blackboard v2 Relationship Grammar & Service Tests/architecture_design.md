Three test files exercise different layers of the Blackboard v2 relationship subsystem:
- `relationship_grammar_internal_test.go` (package `blackboardv2`) drives the in-memory `validateRelationshipEndpoint` validator against a complete matrix generated by `blackboardv2contract.NewHarness().RelationshipCases()`, asserting that every allowed/rejected `(relation, from_type, to_type)` triple is accepted or rejected with a `semantic_validation` error carrying `Details["relation"]`, `Details["from_type"]`, `Details["to_type"]`.
- `relationship_read_internal_test.go` (package `blackboardv2`) injects malformed persisted relationships directly into the SQLite `blackboard_v2_relationships` table via raw `db.Exec` calls and then verifies that both `RuntimeSnapshot` and `ReadCurrent` surface them as `semantic_validation` errors with `Path="relations"` and `Details["violation"]` set to `reason_too_long` or `cycle`; this proves read-time revalidation of on-disk data against the shared grammar.
- `relationship_service_test.go` (package `blackboardv2_test`) uses the public `Service` API through an `evidenceV2Fixture` helper to verify end-to-end behavior: fact derivation (`derived_from` evidence), atomic supersession (versioned replacement, idempotent replay, history preservation, snapshot exclusion of replaced records), confirmed-fact basis protection (rejecting demotions/supersessions that would strand `confirmed` dependents), relationship identity continuity across unrelate/recreate, merge rewrite collision-safe canonical versioning, cycle/self-link rejection for `derived_from` and alias-equivalent pairs, and the 512-byte `Reason` boundary plus normalization rejection of redundant tokenized reasons.

Dependency direction: the two internal tests depend only on the package-private `Service`/`Error` types and helpers; the external test depends solely on the exported `blackboardv2.Service` API and its change-batch schema `semantic-change-batch/v2`. The shared contract matrix lives in `pentest/internal/blackboardv2contract` and is consumed by the grammar test to keep validation logic and its specification in sync.
