## 简介
CyberPenda 是一个“本地优先”的渗透测试代理，由 Go 守护进程、React 仪表盘、沙箱化运行时（Codex/Claude Code/Pi）、项目范围控制、语义记忆平面 Blackboard v2、技能包与运行时扩展、以及 Markdown 报告生成组成。守护进程作为控制面、记忆面、任务生命周期面与报告面；实际工具在选定的运行时环境中执行，而非通过守护进程代理。

本指南面向新用户，目标是帮助你在 15 分钟内完成安装、配置并运行第一个任务。

## 前置要求
- Go：版本以 go.mod 为准（当前为 1.25.x）
- Node.js 20+：用于 UI 构建与本地开发
- Docker 或 Podman：用于沙箱运行时
- 可选：OpenSSL（用于生成 Compose 认证令牌）

说明：
- Go 版本约束见模块定义。
- Node.js 版本要求来自前端工程脚本与依赖。
- 沙箱默认使用 Docker，也可替换为 Podman（通过容器 CLI 参数）。

## 本地开发环境搭建
以下流程将启动后端守护进程与 Vite 前端，并在浏览器中打开仪表盘。

步骤
1) 初始化仓库钩子（推荐）
- 执行一次安装 Git hooks，避免提交时嵌入 UI 不一致导致 CI 失败。

2) 安装 Web 依赖（自动修复）
- 首次运行前会检查并自动修复缺失的 Vite/Rolldown 绑定等依赖。

3) 一键启动开发模式
- 同时启动后端与前端，Vite 会将 /api 与 /health 请求代理到后端。

4) 访问仪表盘
- 打开终端输出的 Vite URL（默认后端监听 127.0.0.1:8787）。

注意事项
- 若后端未就绪，开发脚本会等待健康检查通过后再启动前端。
- 如需自定义沙箱镜像标签，可通过环境变量传入。

## 构建自包含守护进程
适合离线分发或无需 Node.js 环境的场景。

步骤
1) 构建 UI 并嵌入到守护进程
- 先构建 React UI，再复制到守护进程的嵌入路径。

2) 编译守护进程二进制
- 将 UI 资源一并打包进 pentestd 可执行文件。

3) 启动守护进程
- 默认监听地址 http://127.0.0.1:8787。

说明
- 该方式不依赖外部 Node.js 环境，便于直接分发二进制。

## Docker Compose 部署
适合单机快速体验或轻量服务化部署。

步骤
1) 设置认证令牌
- 非回环绑定需要设置 PENTEST_AUTH_TOKEN。

2) 启动服务
- 使用 docker compose up -d 启动。

3) 访问仪表盘
- 打开 http://127.0.0.1:8787/?token=<你的令牌>。

说明
- 容器挂载了宿主 Docker socket，以便应用容器内启动沙箱任务容器。
- 数据持久化到命名卷 cyberpenda-data。
- 镜像默认从 GHCR 拉取，可通过变量覆盖。

## 典型工作流程示例
以下为“15 分钟上手”的标准工作流，涵盖创建项目、定义范围、配置模型提供商、启动任务、查看结果。

1) 创建项目与定义范围
- 在仪表盘中新建项目，明确授权边界与目标资产。
- 范围快照会在任务启动时固化，确保任务行为受控。

2) 配置全局模型提供商
- 在“模型提供商”页面添加一个提供商，填写协议端点与 API Key 环境变量名。
- 刷新模型目录（如支持），选择默认模型或留空由运行时决定。

3) 可选：配置运行时配置文件、凭据绑定、MCP 与技能
- 可在“运行时配置”中选择预设或最小化配置。
- 凭据绑定用于将凭据引用映射到具体来源（环境变量等）。
- MCP 服务器可作为可信接口供运行时读写项目状态。
- 技能包提供通用能力与提示模板。

4) 启动任务
- 在项目仪表盘的“任务启动”中选择运行时插件家族、模型提供商与模型。
- 默认使用沙箱运行器隔离执行。
- 观察活动指示器与时间线，必要时进行引导与继续。

5) 查看结果与报告
- 在“黑板”视图查看实体、关系、证据与发现。
- 生成 Markdown 报告，基于黑板快照输出结构化内容。

说明
- 领域术语请参考上下文文档，确保理解“项目、任务、范围、运行时、黑板”等概念。
- 所有操作均遵循授权范围与策略，避免越权行为。

## Make 目标与常用命令参考
常用目标一览
- make dev：启动守护进程 + Vite 前端（本地开发）
- make build-ui：构建 React UI 并复制到守护进程嵌入路径
- make check-ui-sync：重建 UI 并校验已提交的嵌入是否一致
- make install-git-hooks：启用仓库级 pre-push 钩子
- make build：build-ui + 编译 pentestd
- make build-sandbox-image：构建本地沙箱镜像
- make test / make test-backend：Go 单元与集成测试
- make test-ci：CI 安全测试（无 Docker、无 LLM 凭据）
- make smoke-sandbox-mcp：沙箱 → 守护进程 Blackboard v2 MCP 变更冒烟测试
- make smoke-runtime-tasks：Codex/Claude/Pi 任务冒烟测试（需 Docker + 提供商凭据）
- make clean：清理构建产物

说明
- 开发模式下，Vite 会将 /api 与 /health 代理到 :8787。
- 沙箱镜像标签可通过 SANDBOX_IMAGE 覆盖。
- 测试目标区分 CI 与非 CI 场景，便于流水线与本地调试。

## 常见问题排查
- 端口占用或无法访问
  - 确认守护进程监听地址与端口未被占用；Compose 默认映射 127.0.0.1:8787。
  - 非回环绑定必须设置 PENTEST_AUTH_TOKEN，并通过 ?token= 或 Authorization 头访问。

- 沙箱镜像拉取失败
  - 检查网络与镜像仓库可达性；可通过 SANDBOX_IMAGE 指定本地镜像或私有仓库。
  - 使用 make build-sandbox-image 从源码构建本地镜像。

- 前端依赖缺失或 Rolldown 绑定错误
  - ensure-web-deps 会自动检测并修复；若仍失败，请检查 Node.js 版本与平台原生依赖。

- 模型提供商不可用
  - 确认端点 base_url 与协议匹配，API Key 环境变量已正确设置。
  - 在“模型提供商”页面刷新模型目录，确认模型列表可用。

- 权限与认证
  - 非回环绑定必须设置 PENTEST_AUTH_TOKEN；Dashboard 与 API/MCP 路由均需鉴权。
  - 沙箱容器通过挂载 Docker socket 启动任务容器，注意宿主机安全策略。
